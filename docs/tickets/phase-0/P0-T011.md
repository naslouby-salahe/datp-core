**Deliverable:** `importlinter.ini` at the repository root (the exact filename the repository tree specifies), with `include_external_packages = True` (required for the framework-forbidden contracts to check external packages, not just internal ones) and **seven `forbidden`-type contracts**, one per layer, encoding the complete forbidden-edge closure of the dependency diagram — not just the edges spelled out explicitly, but every edge absent from the diagram (Audit 2's "missing forbidden-import contract is a silent hole" concern, addressed by computing the full closure rather than transcribing only the named prose bullets):
- `domain`: forbidden → all six other layers + all seven frameworks (torch, flwr, sklearn, scipy, pandas, numpy, pydantic).
- `application`: forbidden → config, analysis, infrastructure, composition, cli + all seven frameworks.
- `config`: forbidden → application, analysis, infrastructure, composition, cli + six frameworks (torch/flwr/sklearn/scipy/pandas/numpy — pydantic is config's own legitimate tool, deliberately **not** forbidden here).
- `analysis`: forbidden → config, infrastructure, composition, cli + all seven frameworks (`application` is deliberately **not** forbidden, matching the architecture's narrow documented exception for a framework-free reporting contract).
- `infrastructure`: forbidden → config, composition, cli (frameworks are infrastructure's own job, not forbidden).
- `composition`: forbidden → cli + all seven frameworks directly (frameworks are reached only through infrastructure adapters it constructs).
- `cli`: forbidden → config, application, analysis, infrastructure, domain + all seven frameworks (only `composition` is allowed).

**Invocation note, recorded honestly rather than assumed:** import-linter's built-in auto-discovery only recognizes `setup.cfg`, `.importlinter`, or `pyproject.toml` by filename — **not** `importlinter.ini` (confirmed by reading `importlinter.adapters.user_options.IniFileUserOptionReader.potential_config_filenames` directly in the installed package source). Since the repository tree specification names `importlinter.ini` explicitly, every invocation must pass `--config importlinter.ini` explicitly; bare `lint-imports` with no flag fails with "Could not read any configuration" (verified). This is not a defect — the architecture's chosen filename is honored — but `P0-T014`'s Nox session must invoke `lint-imports --config importlinter.ini`, not the bare form; documented here so that requirement is not lost.
