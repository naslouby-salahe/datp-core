**Environment fact, discovered rather than assumed:** this execution environment genuinely has CUDA hardware available (`nvidia-smi` shows an `NVIDIA GeForce RTX 5060 Ti`; `torch.cuda.is_available()` → `True`), so this ticket's validation exercises real device behavior rather than only a hardware-absence skip path.

**Deliverables:** `tests/support/cuda_lane.py` (`skip_if_cuda_unavailable()` — the `TestDeviceRequirement` skip-policy helper: skip cleanly, never fail, when no CUDA device exists); `tests/integration/cuda/test_serialized_lane_policy.py` with two tests — a real `@pytest.mark.cuda` no-op device check (`torch.zeros(1, device="cuda")`, genuinely runs on the physical GPU) and a `monkeypatch`-based test proving the skip path fires correctly when hardware is (simulated) absent, since the real hardware here cannot be structurally removed to exercise that branch honestly any other way. `tests/conftest.py` gained a `pytest_collection_modifyitems` guard (detailed below).

**Three genuine defects found and fixed during implementation, not merely configured around:**
1. **Package-name collision.** The first collection attempt under `nox -s cuda` failed with `ModuleNotFoundError: No module named 'cuda.test_serialized_lane_policy'` — `tests/integration/cuda/` collided with the real, installed third-party `cuda` namespace package (from NVIDIA's CUDA Python bindings, a transitive dependency of `torch`'s CUDA wheels), because `tests/integration/cuda/__init__.py` existed but its parent `tests/integration/` did not, causing pytest to compute the module's root at `cuda/` instead of `tests/`. **Fixed** by adding `__init__.py` to every test directory that lacked one (`tests/`, `tests/architecture/`, `tests/contract/`, `tests/fixtures/`, `tests/integration/` and its four other subdirectories, `tests/property/`, `tests/support/`, `tests/system/` and its two subdirectories, `tests/unit/` and its four subdirectories) so every test package root chain is unambiguous and explicit, matching the discipline `P0-T005` used for `src/datp_core/`. This revalidates `P0-T009`/`P0-T012`/`P0-T013` (which created some of these directories without `__init__.py`); addenda recorded in each.
2. **Ineffective xdist-detection signal.** The first version of the collection guard checked `session.config.getoption("numprocesses")`, assuming it would read the `-n auto` value. Live-tested: this read back `None`/`dist='no'` in every process — including inside the three spawned xdist workers, confirmed via direct debug instrumentation — because each worker's own config reports itself as non-distributing (it doesn't further distribute). **Fixed** by switching the detection signal to the `PYTEST_XDIST_WORKER` environment variable, which pytest-xdist genuinely sets inside every worker process; confirmed present (`'gw0'`/`'gw1'`/`'gw2'`) via the same debug instrumentation.
3. **Hook-ordering defect.** After fix 2, a real adversarial run (`pytest tests/integration/cuda -n auto`) correctly crashed hard — but then running the **legitimate** `xdist_safe` session (which explicitly excludes `cuda`-marked tests via `-m 'not cuda and ...'`) also crashed, because pytest's own `-m` marker deselection is itself implemented as a `pytest_collection_modifyitems` hook, and without an explicit ordering hint mine could run *before* that deselection, seeing the not-yet-filtered CUDA item. **Fixed** with `@pytest.hookimpl(trylast=True)`, guaranteeing the guard sees the already-marker-filtered item list. Reverified: `xdist_safe` passes clean (`4 passed`), `cuda`/`serial`/`resource_intensive` all pass clean, and the adversarial rejection still fires.
