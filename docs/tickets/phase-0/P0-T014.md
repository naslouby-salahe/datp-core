**Two genuine, serious defects found and fixed during implementation, not merely configured around:**
1. **PATH resolution defect.** The first version of `noxfile.py` used `venv_backend="none"` with bare command names (`session.run("ruff", ..., external=True)`). Testing each session individually revealed `lint-imports` was "not found," and every pytest-invoking session failed with `ModuleNotFoundError: No module named 'hypothesis'` — even though Hypothesis is correctly installed in `.venv`. Root cause, confirmed directly: `which pytest` resolved to `/home/naslouby/.local/bin/pytest` and `which pyright` resolved to an unrelated Node/nvm-distributed `pyright` binary — both entirely different installations from this project's pinned `.venv` tools, found first on this machine's shell `PATH`. The earlier "successful" `lint`/`format`/`typecheck` session runs during this same investigation were **silently validating against the wrong tools** before this was caught. **Fixed** by routing every tool invocation through `sys.executable -m <module>` instead of a bare command name — this pins each call to the exact interpreter/environment that invoked Nox, independent of shell `PATH` contents, with the reasoning stated as a code comment (the actual engineering rule, not a citation). Reverified every session afterward, confirming from the printed command line that `.venv/bin/python3` is genuinely the interpreter used.
2. **`noxfile.py` outside Pyright's checked scope.** `pyright noxfile.py` passed when invoked directly, but the repository-wide `pyright` command (what later gates actually run) silently skipped it because `[tool.pyright].include` only listed `["src", "tests"]`. Fixed by adding `"noxfile.py"` to `include` (owned by `P0-T008`; addendum recorded there), satisfying B8's explicit requirement that `noxfile.py` pass strict typing as part of the tracked source tree.

**Marker additions (T014's own session-selection wiring, not a reopening of `P0-T009`):** `serial` and `resource_intensive` are Nox session concepts, not `TestSuiteKind` members — no marker for either existed after `P0-T009` (which registered only the nine `TestSuiteKind`-based markers). Without them, the `serial` and `resource_intensive` sessions would have no way to select anything. Added both to `pyproject.toml`'s marker list with a comment distinguishing them from the `TestSuiteKind` set.
