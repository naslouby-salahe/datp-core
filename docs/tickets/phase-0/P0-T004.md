## Completion record

**2026-07-14 — dependency-lock discipline established.** `P0-T003` verified `DONE` first. Lifecycle: `NOT_STARTED` → `IN_PROGRESS` → `IN_REVIEW` → `DONE`.

**Committed lock.** `uv.lock` (110 packages resolved from the groups `P0-T003` declared) is committed at the repository root, generated by `uv lock` against the Python `==3.12.*` constraint.

**Regeneration rule (deterministic, explicit — not "as needed"):**
1. The committed `uv.lock` is the reproducibility anchor, not the abstract constraint set in `pyproject.toml`. Routine dependency work (`uv sync`, plain `uv lock`) must be run **with the committed lock present**; in that mode `uv lock` is idempotent — verified by running it twice against the already-committed lock and confirming byte-identical SHA-256 (`627c61cc7350638dce59b3e7635b65a59b0945cccf05a869106a845736350e11`) both times.
2. The **only** sanctioned way a pin changes is an explicit `uv lock --upgrade` (whole-graph) or `uv lock --upgrade-package <name>` (single package), run deliberately, with the resulting `git diff uv.lock` reviewed and the rejected-library scan (`P0-T003`'s method) rerun before the new lock is committed.
3. `uv.lock` is never hand-edited. No stale/legacy lock is retained alongside it.

**Genuine adversarial finding (Audit 3), recorded rather than hidden:** deleting `uv.lock` entirely and re-resolving from scratch (simulating "the committed lock is lost") produced a **different** hash — `ee8c221664e1957af6c231b94cafaaa33ca395bf28cbec06585776bc94364baa` vs the committed `627c61c...`. Root cause, diffed directly: (a) `ruff`'s floating `>=0.6` constraint resolved to a newer patch release (`0.15.20` → `0.15.21`) that was published to PyPI in the few minutes between the two test resolutions — none of the dev-tooling constraints in `P0-T003` are exact-pinned, only the 8 named numerically/order-sensitive scientific libraries are (correctly, per Architecture §9.5's own scope); (b) a small number of platform-marker representations for CUDA-related transitive packages (`cuda-pathfinder`, `nvidia-cublas-cu12`, `nvidia-nvjitlink-cu12`) differed cosmetically between a from-scratch resolve and an existing-lock-present resolve — a `uv` resolver-internal variance, not a version drift. **Disposition:** this is why the regeneration rule above anchors reproducibility to the *committed lock file*, not to "re-resolve the constraints and hope for the same answer" — the committed lock was restored to the `P0-T003`-reviewed content (hash `627c61c...`, confirmed via `sha256sum` and `git diff --stat uv.lock` showing the working tree matches the intentional lock) and `uv sync --all-groups --all-extras` reconfirmed a clean install from it (`Resolved 110 packages in 1ms`, `Checked 108 packages in 1ms`). No dependency version was silently changed by this ticket.
