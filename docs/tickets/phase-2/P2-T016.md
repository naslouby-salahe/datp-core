## Completion — 2026-07-17T23:35:00Z

**Idempotent-rerun finding.** `PolicyEvaluator.evaluate()`, `ClientEvaluationResult`, `FleetDispersionResult`, and `cv_outcome()` (`application/stages/evaluate_policy.py`, `domain/evaluation/operating_points.py`) were already implemented and substantially tested from an earlier undocumented commit. Verified genuine: confusion counts and FPR/TPR are computed correctly with concrete numeric assertions (`tests/unit/application/test_stage_services.py::test_policy_evaluator_uses_the_committed_test_aggregate_and_counts`); a calibration set passed in the test-score position is rejected at runtime (`test_policy_evaluator_rejects_a_calibration_score_set_in_the_test_position`); zero-mean CV degeneracy is a typed `UndefinedCvResult`, never an exception (`tests/unit/domain/test_operating_point_specifications.py::test_zero_mean_cv_is_a_typed_outcome_not_an_exception`); confusion-count recomputability (`benign_test_count == TN + FP`, `attack_test_count == TP + FN`) is a genuine Hypothesis property test (`tests/property/test_operating_point_specifications.py::test_client_evaluation_preserves_recomputable_confusion_count_totals`). Accepted as prior work, not recreated.

**Real gap found and closed.** No test proved that evaluating two different core-ladder policies (B1, B2) against the *same* `EligibleClientSet`/`TestScoreArtifactSet` inputs actually shares that population rather than each policy computing its own — the architectural guarantee exists by construction (`EligibleClientSet`/`TestScoreArtifactSet` are caller-supplied `EvaluatePolicyRequest` fields, never computed internally by `PolicyEvaluator`), but this ticket explicitly requires a unit test proving it ("Paired policies share one eligible set"). Added `tests/unit/application/test_stage_services.py::test_paired_policies_reuse_one_eligible_set_and_test_score_set`, which evaluates B1 and B2 from the identical `eligible_client_set`/`score_set` objects and asserts both results carry the identical eligible-set identity and identical per-client FPR/TPR (since both draw from the same underlying committed test aggregate).
