**Real defect found and fixed.** `B4ClusteringSpec.__init__` sourced `n_init` (10), `max_iter` (300), and `scikit_learn_version` ("1.9.0") from bare module-level `Final` Python constants (`CANONICAL_KMEANS_N_INIT`, `CANONICAL_KMEANS_MAX_ITER`, `PINNED_SCIKIT_LEARN_VERSION`), even though `configs/scientific/thresholds.yaml`'s `b4_profiles` canonical entry already declared the same `n_init`/`max_iter`/`scaler`/`algorithm`/`fingerprint_fields` values — a genuine "duplicated defaults, not loaded from the validated configuration boundary" violation of this ticket's own explicit acceptance criterion ("no bare `__post_init__` constant may own them"). `scikit_learn_version` was not even present in the schema/YAML at all. Fixed:
- Added `scikit_learn_version: Annotated[str, Field(pattern=r"^\d+\.\d+\.\d+$")]` to `config/schemas/catalog.py::B4ClusteringProfileConfig`; added `scikit_learn_version: "1.9.0"` to both `b4_profiles` entries in `configs/scientific/thresholds.yaml` (verified against the real pinned dependency: `pyproject.toml` pins `scikit-learn==1.9.0`, and the installed environment reports the same via `importlib.metadata.version`).
- Added `domain/thresholding/clustering.py::CanonicalB4ClusteringProfile` — a frozen, `__post_init__`-locked value object (fingerprint fields, scaler, scaler_fit_scope, algorithm, n_init, max_iter, scikit-learn version) that rejects anything but the true canonical values, exactly mirroring the existing `B0PooledThresholdSpec` locked-value pattern.
- Added `config/mapping/catalog.py::map_canonical_b4_clustering_profile_config`, and wired it into `map_profile_catalogue` as `ProfileCatalogueSpec.canonical_b4_profile`, selecting the `is_canonical=true` entry from `thresholds.b4_profiles`.
- Changed `B4ClusteringSpec.__init__` to require a `profile: CanonicalB4ClusteringProfile` parameter (no default — a default would have reintroduced the same hidden-default defect one layer up) and derive `fingerprint_fields`/`scaler`/`scaler_fit_scope`/`algorithm`/`n_init`/`max_iter`/`scikit_learn_version` from it. `cluster_count` stays `CANONICAL_CLUSTER_K`, a pure code-level constant, per this ticket's own explicit instruction ("K=3 remains a code-level invariant... non-overridable"). `CANONICAL_KMEANS_N_INIT`/`CANONICAL_KMEANS_MAX_ITER`/`PINNED_SCIKIT_LEARN_VERSION` are retained solely as the comparison baseline inside `_verify_specification`'s existing tamper-detection check (which defends against a frozen `B4ClusteringSpec` instance corrupted via `object.__setattr__` after construction — a scenario no amount of `__post_init__` validation at construction time can catch, and the one the existing test already exercises); they no longer supply the spec's own field values at construction.
