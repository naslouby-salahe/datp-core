### Completion evidence — 2026-07-15

- Dependencies verified: `P2-T006` `DONE`, `P1-T020` `DONE` (`docs/tickets/TICKET_STATUS.md:71,183`).
- Implementation: added `NBaIoTPreprocessorFitter` and `NBaIoTProcessedSplitMaterializer` (`infrastructure/data/nbaiot_preprocessing.py`), plus new domain types `FeatureStatistics`/`ClientFeatureStatistics`/`FittedPreprocessorManifest` (`domain/data/preprocessing.py`). `is_authorized_nbaiot_policy`/`_validate_locked_preprocessing_policy` lock the recovered P2-T001 preprocessing semantics (`STANDARD` / `PER_CLIENT_TRAIN` / `EXACT_TWO_PASS`); any other `PreprocessingSpec` combination raises `PreprocessingError` before any I/O happens.
  - **Fit path (`NBaIoTPreprocessorFitter.fit`)**: for each device, re-derives the client roster via the existing `sorted_device_directories` (same pattern as P2-T004/P2-T005/P2-T006), recomputes the client's TRAIN/GAP/CALIBRATION/GAP/TEST boundaries by reusing P2-T006's now-public `benign_split_boundaries`/`row_range_roles`, extracts **only** the TRAIN-role row ranges into a scratch parquet file (`_extract_train_raw`, feature-columns-only, with a row-order checksum), and fits exact per-feature mean/variance over that TRAIN-only file with the existing, already-tested `TwoPassNumericPreprocessor` (Phase 1 component; sklearn's `StandardScaler` deliberately not used here since it requires loading the full array into memory, contradicting the two-pass streaming design already established for this concern). CALIBRATION and TEST rows are never read into the fitter — enforced structurally by `row_range_roles` filtering to `SplitRole.TRAIN` before any row reaches the scratch file, not by a downstream check. The manifest is persisted as a content-addressed `FITTED_PREPROCESSOR` artifact; `FittedPreprocessorResult.training_row_order_checksum` aggregates all per-client TRAIN checksums (sorted by client id for determinism) via `blake3`.
  - **Transform path (`NBaIoTProcessedSplitMaterializer.materialize`)**: reads back the persisted `FittedPreprocessorManifest`, re-derives the same client roster and boundaries, and streams each client's materialized source file once, writing normalized TRAIN/CALIBRATION/TEST parquet files in a single pass (`_write_all_roles`/`_write_batch_roles`/`_write_normalized_segment`) using the fitted per-client statistics (`(x - mean) / sqrt(variance)`, with a `variance == 0` guard emitting `0.0` rather than dividing by zero). Feature column order is preserved explicitly (`_normalized_batch` builds arrays in `feature_columns` order, not source-file column order). Each non-empty role file becomes a `PROCESSED_SPLIT` `ArtifactRef`, content-hashed from the written file.
  - Design decision (not dictated by Architecture, documented per B2): `ProcessedSplitResult.split_manifest_identity` is set to `request.split_manifest.split_identities.training.split_identity` — the port carries three split identities (training/calibration/test) but `ProcessedSplitResult` has a single identity field, and TRAIN was chosen as the representative identity since it is the fit-authorizing split. Revisit if a later ticket's consumer expects a different one.
  - Design decision: both adapters take `raw_root`/`materialized_root`/(`processed_root`/`scratch_root`) via constructor injection and independently re-derive the client roster, continuing the pattern established in P2-T004–P2-T006 because the abstract `FitPreprocessorRequest`/`MaterializeProcessedSplitsRequest` ports do not carry concrete filesystem roots.
