### Completion evidence — 2026-07-15

- Dependencies verified: `P2-T003` `DONE`; `P1-T042` `DONE` (`docs/tickets/TICKET_STATUS.md:210`).
- Implementation: `NBaIoTChunkedSourceAdapter` in `src/datp_core/infrastructure/data/nbaiot_source.py`. For a given device, streams every source CSV (in the same deterministic device/label/filename order established by P2-T003's `device_csv_files`, now made a shared, reusable function rather than duplicated) with bounded-memory `pyarrow.csv.open_csv`, injects a `source_label` column per row (benign/gafgyt/mirai, from the filename/directory convention, since downstream splitting (P2-T006) needs this to separate benign-only calibration from held-out benign+malicious test), and writes one streamed, bounded-memory Parquet file per device via `pq.ParquetWriter`. Deterministic source-row identity reuses the existing shared `update_row_order_checksum` mechanism (not a new algorithm) rather than inventing a parallel identity scheme.
- **Real defect found and fixed in existing Phase 1 infrastructure, not merely in new code:** `update_row_order_checksum` (`src/datp_core/infrastructure/data/streaming.py`) hashed the batch's column names on *every* batch call, not once per stream. Since chunk/batch boundaries vary with configured batch size, this made the checksum batch-size-dependent — directly contradicting this ticket's own acceptance criterion ("chunked read equals reference") and Architecture's stable-row-order-checksum invariant. Confirmed by a failing test before the fix (different `csv_block_bytes` producing different checksums for identical row content) and confirmed passing after removing the per-batch name-hashing (schema consistency is already independently enforced by an explicit check at every real call site — `materialization.py`'s `_write_batch` and this ticket's own materializer both already reject a schema change mid-stream — so this fix does not weaken any existing guarantee). Added a direct regression test at the shared `streaming.py` level (`tests/integration/data/test_pyarrow_streaming.py::test_row_order_checksum_is_independent_of_the_batch_size`), not only an indirect one through this ticket's own adapter, since the defect was in shared infrastructure other tickets also depend on.
- **Second defect found and fixed during property-test authoring:** an unhandled raw `pyarrow.lib.ArrowInvalid` could escape the adapter boundary when a configured CSV block size was too small to hold even one row/header line — violating this codebase's typed-error discipline (no raw third-party exception crossing an adapter boundary). Fixed by catching `pyarrow.ArrowInvalid` around the CSV read/iterate path and re-raising as a typed `DatasetError` with an actionable message; added `ArrowInvalid` to the hand-maintained `typings/pyarrow/__init__.pyi` stub (with a `per-file-ignores` entry in `pyproject.toml` for Ruff's `N818`, since the stub must mirror pyarrow's actual exception class name exactly and cannot rename it to satisfy internal naming convention).
- Extended `typings/pyarrow/__init__.pyi` (the `pyarrow.csv` stub itself was already added by `P2-T003`) with `Table.schema`/`Table.num_rows`, `RecordBatch.from_arrays`/`num_columns`, `DataType`, `array()`, `string()`, and `ArrowInvalid`, all needed for the label-column injection and error handling this ticket adds.
- Real read-only materialization was run against the actual N-BaIoT source (`data/raw/N-BaIoT/Ennio_Doorbell`, the smallest of the 9 real devices — benign + 5 gafgyt files, no mirai) at two very different CSV block sizes (64KiB and 16MiB): both produced identical `row_count=355500` and an *identical* `row_order_checksum` (`f80313ec...`), directly proving chunked-vs-reference equivalence and deterministic row identity on real data, not only the synthetic fixture. Output written to session scratchpad only (large per-device Parquet intermediates are not committed to the repository, consistent with `.gitignore`'s treatment of processed/output artifacts — only P2-T003's small manifest JSON files are canonical committed deliverables).
- Acceptance criteria: deterministic source-row identity — PASS (real-data and synthetic proof, both block-size-independent). Chunked read equals reference — PASS (same mechanism, now fixed to be genuinely batch-boundary-independent). Bounded memory — PASS (CSV reads stream via `pyarrow.csv` block-size-bounded batches; Parquet writes stream via `ParquetWriter.write_batch` per batch; no `pandas.read_csv`/whole-file load anywhere). No scientific run — PASS (no training/scoring/partitioning/splitting occurred; output is an intermediate materialization only).
- Required property test (Part A item 26, "Row-identity determinism"): `tests/property/test_nbaiot_source_row_identity.py`, Hypothesis-driven over random row counts (1–50) and random CSV block sizes (256B–4KiB) vs. a fixed large block size, asserting identical row counts and checksums. Plus the new shared-infrastructure regression test in `test_pyarrow_streaming.py`. All pass.
