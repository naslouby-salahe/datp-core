13. **Objective.** Implement the scalar value objects of Architecture §7 with `__post_init__` validation: `ThresholdPercentile`/`FprTarget`/`ConfidenceLevel`/`CoverageRatio`/`EligibilityCoverage`/`ConformalCoverage`/`Probability` as canonical `Decimal` types (quantized exactly once at construction), and every float-wrapping rate/score (`FalsePositiveRate`, `TruePositiveRate`, `AuRocScore`, `ThresholdValue`, `DirichletAlpha`, `ShrinkageWeight`) rejecting `NaN`/infinity in addition to its range check; enforce `FprTarget == 1 − q` on canonical Decimals; keep every probability-like type mutually non-interchangeable.
14. **Why this ticket exists.** A `NaN` slipping into a fingerprinted field (for example from an empty-slice mean) would make two otherwise identical specifications compare unequal, because `NaN != NaN` in IEEE 754, silently breaking stage-identity reuse and deduplication (Architecture §7.3) — rejecting non-finite values at construction removes this failure mode entirely rather than relying on downstream code to notice. Decimal canonicality makes `q` and `1 − q` exact and reproducible regardless of which computation path produced them, so the `FprTarget == 1 − q` invariant never depends on floating-point rounding order. Distinct, non-interchangeable types prevent a confidence level from being passed where an FPR target or coverage ratio is expected — a class of defect that a shared `float` type could never catch at compile time.
