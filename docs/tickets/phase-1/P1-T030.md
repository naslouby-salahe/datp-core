13. **Objective.** Implement `domain/errors.py` with `DatpCoreError` as the root and every error family named in field 12, each carrying typed context and an explicit `FailureDisposition`.
14. **Why this ticket exists.** A scientific pipeline that fails silently or with a bare-string exception loses exactly the information needed to distinguish a retryable transient condition from a run-blocking scientific defect. Every error family here carries a `FailureDisposition` (`RUN_BLOCKING`, `STAGE_BLOCKING`, `RETRYABLE_TRANSIENT`) so the lifecycle machinery built in P1-T039 can react correctly without inspecting exception message text. This ticket is also the single place where the architecture's explicit rejection of a `Result[T, E]` monad (Architecture §20.3) is honored — errors are raised, not returned, throughout the codebase.
15. **Authority references.** Architecture §20.1 (complete error-family catalogue), §20.2 (disposition semantics), §20.3 (no `Result[T, E]` monad — exceptions are the sole error-propagation mechanism).

20. **Inputs.** Architecture §20.1, §20.2, §20.3.
21. **Allowed scope.** `domain/errors.py`.
22. **Forbidden actions.** No bare-string exception raised anywhere this hierarchy should be used instead; no one-off exception class created per error message (each family is a class, not each message); no exception swallowing (a caught error must be re-raised, translated to another typed error, or handled explicitly — never discarded); no monadic outcome type competing with this hierarchy.
