13. **Objective.** Implement `infrastructure/persistence/hashing.py`: blake3 content hashing as the default for content addressing (score arrays, Parquet files, state dicts) and SHA-256 via `hashlib` where a cryptographic guarantee is specifically required.
14. **Why this ticket exists.** This ticket was split out of a former combined persistence mega-ticket because hashing is a narrow, independently testable utility with its own performance and correctness properties, entirely distinct from path resolution (P1-T054), serialization (P1-T056), atomic commit (P1-T057), bundle commit (P1-T058), and locking (P1-T059). blake3 is five to ten times faster than SHA-256 on large arrays and is the default for routine content addressing (Architecture §4.1); SHA-256 remains available only for the narrow case where a cryptographic guarantee is specifically required, never as the default.
15. **Authority references.** Architecture §4.1 (blake3/hashlib role assignments), §9.5 (blake3 for content hashes; SHA-256 where required), §13.4 (fingerprint computation uses blake3 over canonical typed tuples, never raw content — this ticket provides the shared blake3 primitive both fingerprinting and artifact hashing use).
16. **Dependencies.** P1-T054.
17. **Blocks.** P1-T057, P1-T058, P1-T042, P1-T043, P1-T046, P1-T070.
18. **Downstream consumers.** P1-T057 (verifies content hash before commit), P1-T058 (bundle member-hash verification), and every adapter that persists a content-addressed artifact.
19. **Preconditions.** P1-T054 is `DONE`.
20. **Inputs.** Architecture §4.1, §9.5, §13.4.
21. **Allowed scope.** `infrastructure/persistence/hashing.py`.
22. **Forbidden actions.** No use of a content hash (blake3 or SHA-256) to derive a logical `ArtifactId` — identity is always deterministic from type/key/stage/namespace, never from raw content bytes; no hashing of raw floats without first passing through the canonical quantization step the domain fingerprint model defines (Architecture §7.3, §13.4).
23. **Required configuration or NONE.** NONE.
24. **Implementation responsibilities.** Implement a streaming blake3 hash function suitable for large arrays and files; implement a SHA-256 path reserved for the cryptographic-guarantee case; expose both behind narrow, explicitly named functions — never a generic `hash(data, algo)` dispatcher.
25. **Required unit tests.** Identical bytes hash identically across repeated calls and across a chunked-vs-whole-file streaming comparison; two different byte strings hash differently across the test corpus (no trivial collision); the SHA-256 path is used only where explicitly requested, never as a silent default.
26. **Required property tests.** Hash output length and encoding remain stable across arbitrary byte-string inputs within the declared size bounds.
27. **Required contract tests.** NONE.
28. **Required architecture tests.** `hashing.py` exposes no generic `compute(algo, data)` method.
29. **Required integration tests.** A large synthetic array or Parquet file hashes identically whether read whole or in chunks.
30. **Required CUDA tests.** NONE.
31. **Required system/end-to-end tests.** NONE.
32. **Validation commands.** Unit lane; property lane; integration lane (persistence).
