13. **Objective.** Implement `config/schemas/{scientific,execution,artifacts,reporting}.py` Pydantic v2 boundary schemas with discriminated unions for policy-specific fields, one schema owner per field, targeting the three fingerprint categories.
14. **Why this ticket exists.** External configuration is validated only at the `config` boundary: Architecture §4.1 confines Pydantic v2 to the `config` boundary and states explicitly that no Pydantic model crosses into `application`, `domain`, or `analysis` — this ticket is the sole owner of every Pydantic v2 model in the repository, split into `config/schemas/scientific.py`, `execution.py`, `artifacts.py`, and `reporting.py` by the three fingerprint categories of Architecture §11 (scientific, fingerprinted; execution, recorded and only-where-output-affecting; environment, recorded only) plus the artifact/reporting sections that configuration also validates. Every configuration field has exactly one schema owner (§11) so no field can be validated two different ways in two different schemas. Policy-specific fields — the threshold-construction variant, the statistical procedure, and others — are carried through Pydantic v2 discriminated unions with an explicit tag field, never inferred from which optional fields happen to be populated, because inferring a variant from field presence is exactly the kind of implicit branching Architecture §11.4 forbids (no `use_full_pooled_variance` or tie-break boolean; full pooled variance and larger-k ties are algorithm definitions, not configuration toggles). `EvaluationConfig.primary` is locked to `CV_FPR` and a confirmatory experiment's `StatisticalConfig` is locked to `BCA_BOOTSTRAP`/0.95/ten paired seeds at the schema layer itself (§11.4), so an invalid confirmatory statistics shape is rejected before it ever reaches a domain-mapping function in P1-T032.
15. **Authority references.** Architecture §4.1 (Pydantic v2 accepted-library confinement to the `config` boundary only), §11 (two senses of "configuration", one schema owner per field, three fingerprint categories), §11.1–§11.4 (scientific/execution/environment schema ownership and the mapping/validation rules: `EvaluationConfig.primary` locked to `CV_FPR`, confirmatory `StatisticalConfig` locked to `BCA_BOOTSTRAP`/0.95/ten, no `use_full_pooled_variance`/tie-break boolean, no hidden defaults), §3.2 (`config →` domain only; `config → infrastructure` and `config → cli` are forbidden dependency directions).
