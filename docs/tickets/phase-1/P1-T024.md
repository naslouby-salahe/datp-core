13. **Objective.** Implement the closed `ThresholdConstructionSpec` union of the nine FedAvg-derived variants listed in field 12, each carrying the explicit `ThresholdConstructionKind` tag; `ThresholdSuiteSpec` holding an ordered tuple of these variants evaluated over one shared calibration score set; `ThresholdAssignment` recording the exact `CalibrationScoreArtifactId` a threshold was derived from. 14. **Why this ticket exists.** This union is the concrete realization of Architecture §6.10's rule that a specification with more than one variant carries an explicit enum tag, never an inferred one: nine structurally similar threshold constructions (differing in which optional fields they populate) would otherwise invite exactly the "which optional fields are compatible" defect class the architecture forbids. `ThresholdSuiteSpec` not owning scoring configuration matters because it keeps `ScientificProtocolSpec.scoring` the single authority for how scores were generated — a threshold suite that could redeclare its own scoring parameters would let B1–B4 silently diverge on the score set they read from.
