14. **Why this ticket exists.** Statistics is SciPy-backed (Architecture §4.1: SciPy confined to `infrastructure`, required for BCa bootstrap, Wilcoxon, Spearman, Jensen–Shannon) yet the application layer must be testable without SciPy — `StatisticalProcedureRunner` is the port behind which that dependency hides, taking one named `RunStatisticalAnalysisRequest` and returning one named `StatisticalAnalysisResult` (Architecture §12.2, §9.7). Rendering a table, figure, or wording block is an `infrastructure` concern reached only through `ReportRenderer.render(RenderReportArtifactRequest) -> RenderedReportArtifact` (Architecture §12.2); `analysis` itself never touches a renderer or a repository (Architecture §22.1, §22.3's guarantee that a renderer only chooses a serialization format and never alters a scientific value). Events are typed and never the scientific source of truth: Architecture §19 states plainly that logs are diagnostic and that scientific values live in persisted artifacts, never in a log line — the application emits typed events through `EventSink.publish(StructuredEvent) -> None` and never formats a string for a console itself (§19.2). The `EventSink`/`StructuredEvent`/`EventContext`/discriminated-detail-union vocabulary is already canonically defined by P1-T051 in `application/ports/telemetry.py`; this ticket does not redefine that vocabulary. It re-exercises and exposes it for the statistics and reporting ports' own consumers — a stage function that needs to publish a `STAGE_HEARTBEAT` or a determinism event while also invoking `StatisticalProcedureRunner` or `ReportRenderer` — so P1-T051 remains the vocabulary's sole owner while `application/ports/telemetry.py` is completed at the point this ticket's consumers actually need it wired alongside the statistics/reporting ports.
